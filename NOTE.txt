ยกสถานการณ์ที่ Kubernates เข้ามาแก้ไขปัญหา
[
1. auto restart container
2. auto scale service ตาม triffic ด้วยคำสั่ง kubectl scale
3. rolling update -> pod ใหม่ขึ้น pod เก่ายังใช้งานได้ -> zero downtime
4. สร้างช่องทางการสื่อสารระหว่าง container
5. กระจาย container ตาม server ที่มี cpu/memory อัติโนมัติ
]
Use Case ยอดนิยมที่องค์กรเลือกมาใช้ Kubernates
[
รันแอปจำนวนมากแบบอัติโนมัติบนหลายเครื่อง/หลาย Cloud
1. ต้องการ scale
2. ต้องการ downtime น้อย
3. triffic ไม่คงที่ -> ต้องการ auto scale
4. ใช้ multi cloud หรือ hybrid on-premise + cloud
5. batch workload -> ปิดเมื่อจบงาน
]
Use Case ที่ไม่เหมาะที่จะนำ Kubernates มาใช้
[
1. แอปเล็ก / traffic น้อย
2. legacy app ที่ผูกติดกับ os
]

เราจะ Auto scale ตามโหลดอย่างไร เพราะตัวอย่างมันเป็น manual คำสั่งหมดเลย
[ใช้ HorizontalPodAutoscaler (HPA)
HPA จะวนตรวจทุก ~15 วินาที แล้วคำนวณว่าต้องใช้กี่ Pod ตามสูตรคร่าว ๆ:
เช่น มี 2 Pod, CPU จริงเฉลี่ย 100%, เป้าหมาย 50% → ต้องการ 4 Pod
]
ไฟล์ vote-hpa.yaml ทำงานอย่างไร
[
สร้าง HPA ชื่อ vote
กำหนดเป้าหมายที่จะควบคุมที่ Deployment ชื่อ vote
กำหนดช่วงจำนวน Pod
รักษา 50% ของ CPU Request ต่อ Pod
]
จากไฟล์ vote-deployment.yaml ส่วนนี้ทำงานอย่างไร สัมพันธ์กัน vote-hpa.yaml อย่างไร
        resources:
          requests:
            cpu: 100m #0.1 core
            memory: 64Mi
          limits:
            cpu: 250m
            memory: 128Mi 
[แต่ละ Pod ใช้อย่างน้อย 0.1 core / 64MiB ถ้าจะวาง Pod ใหม่ใน Node (ไม่ใช่ค่าใช้จริง) และได้ไม่เกิน 0.25 core / 128MiB
HPA จะพยายามรักษาให้ได้ 0.05 core / Pod หากสูงกว่าจะเปิด Pod ใหม่ ถ้ายังเหลือ Resource
]
ถ้า Pod ถูกสร้างใหม่ แล้วข้อมูลหาย จะแก้ปัญหานี้อย่างไร ตาม Best Practice
[ใช้งาน PersistentVolumeClaim หรือ StatefulSet ส่วน Production จะใช้ cloud managed database]
